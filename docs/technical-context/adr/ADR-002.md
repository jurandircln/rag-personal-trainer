# ADR-002: Qdrant como Vector Store

**Data:** 2026-03-15
**Status:** Aceito
**Autores:** Time Jarvis

---

## Contexto

O projeto Jarvis precisa de um vector store capaz de:
- Realizar busca por similaridade coseno em alta velocidade
- Armazenar metadados por ponto (fonte, pagina, chunk_id)
- Operar localmente sem custo por chamada
- Persistir dados entre reinicializações

---

## Decisão

Adotar **Qdrant** rodando localmente via Docker como vector store do Jarvis.

- Porta: `6333` (HTTP) / `6334` (gRPC)
- Persistência: `./data/qdrant_storage`
- Setup: `docker-compose.yml` na raiz do projeto

---

## Consequências

### Positivas
- **Sem custo:** operação totalmente local
- **Metadados flexíveis:** cada ponto pode ter payload arbitrário
- **Dashboard embutido:** interface web em `http://localhost:6333/dashboard`
- **Cliente Python maduro:** `qdrant-client` com suporte a upsert em lote

### Negativas / Riscos
- **Dependência de Docker:** ambiente de desenvolvimento requer Docker
- **Escala local:** adequado para uso single-user; multiusuário exigiria instância gerenciada
- **Sem sincronia automática:** reindexação manual ao atualizar documentos

---

## Alternativas Descartadas

| Opção | Motivo da Rejeição |
|---|---|
| Pinecone | Cloud pago, dependência externa |
| ChromaDB | Menos maduro, API em evolução frequente |
| pgvector | Overhead de Postgres para caso de uso puramente vetorial |
| FAISS | Sem servidor, sem persistência nativa entre sessões |
