# ADR-004: Arquitetura Modular sem Camada REST

**Data:** 2026-03-15
**Status:** Aceito
**Autores:** Time Jarvis

---

## Contexto

O projeto Jarvis precisa definir como a interface Streamlit se comunica com os módulos de retrieval e generation. As opções são: importação direta de módulos Python, ou expor os módulos como uma API HTTP.

O uso atual é single-user local — um personal trainer rodando a aplicação em sua própria máquina.

---

## Decisão

Adotar **arquitetura modular sem camada REST**: o Streamlit importa `SemanticSearcher` e `RAGGenerator` diretamente como módulos Python, sem intermediário HTTP.

```python
# Exemplo de uso no app.py (futuro)
from retrieval.searcher import SemanticSearcher
from generation.llm import RAGGenerator
```

---

## Consequências

### Positivas
- **Sem overhead:** sem serialização/deserialização HTTP para chamadas locais
- **Simplicidade:** menos código, menos superfície de falha
- **Deploy simples:** um único processo Python (Streamlit) gerencia tudo
- **Debug mais fácil:** stack trace direto, sem camadas de rede

### Negativas / Riscos
- **Não escalável para multiusuário:** sem servidor HTTP, não há como servir múltiplos clientes concorrentes
- **Acoplamento:** interface acoplada diretamente aos módulos de negócio

### Mitigações
- Se o projeto escalar para múltiplos usuários, uma ADR nova deve ser criada para adicionar camada REST (FastAPI ou similar)
- A abstração via classes (`SemanticSearcher`, `RAGGenerator`) facilita eventual wrapping em endpoints HTTP

---

## Alternativas Descartadas

| Opção | Motivo da Rejeição |
|---|---|
| FastAPI | Overhead desnecessário para uso single-user local |
| gRPC | Complexidade excessiva para o caso de uso atual |
| Flask | Mesma justificativa do FastAPI, sem vantagem adicional |
